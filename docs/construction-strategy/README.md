# AI事業OS 構想フロー（建設起点 → 業界展開型）

## 概要

建設業界を起点に「課題クラスター型」のビジネスモデルを構築し、検証済みの型を他業界へ横展開する戦略フレームワーク。

スコア化は行わず、**支払い証拠ベースのPASS/FAIL判定**で課題を選別する。

## システム前提（Phase 0）

| 項目 | 方針 |
|------|------|
| アーキテクチャ | 共通基盤 + 業界別アダプタ |
| 共通基盤 | 使い回す（SNS生成、LP生成、FAQ、一次返信、ログ収集） |
| 業界別アダプタ | 差し替え可能にする（業界固有の用語・課題・顧客像） |
| 実行方針 | 実行は少数、調査は広く |
| 成果判定 | 「問い合わせ数」ではなく「商談化・受注」で見る |

## 運用の核心ルール

### A. 調査は広く、実行は狭く
- 調査対象は広くてよい
- 実行対象は同時に少数に絞る

### B. 実行単位は「顧客1人」ではなく「課題クラスター」
- 1顧客 = 1SNS はやらない
- 近い課題は1導線でまとめる

### C. コンテンツ量産は受注前提ではなく検証前提
- 最初から1000記事やらない
- 当たってから増やす

### D. AIの役割は量産だけではない
- 調査 → 生成 → 一次対応 → ログ整理 → 改善提案 まで回す

### E. 価値は「業界別の設計」にある
- 共通基盤は安く作る
- 高単価化は業界別アダプタで作る

## 既存システム（MarketProbe V2）との関係

本戦略は既存のMarketProbe V2パイプラインを**建設業界に特化して運用する**形で活用する。

| 本戦略のフェーズ | MarketProbe V2の対応機能 |
|----------------|------------------------|
| Phase 2-5（調査） | `A0`（マイクロマーケット生成）を建設業界に限定して実行 |
| Phase 6-7（証拠確認） | `A1q`/`A1d`（証拠ベースゲート）がそのまま適用可能 |
| Phase 8-9（競合調査） | `C`（競合分析20社×7URLタイプ）を課題単位で実行 |
| Phase 10（オファー化） | `0`（3クイックオファー生成）を活用 |
| Phase 11（検証パッケージ） | LP生成 + SNS投稿 + フォーム営業の既存パイプライン |
| Phase 12（実行） | `2`（SNS投稿）+ `3`（フォーム営業）+ `6`（広告監視） |
| Phase 13（記録） | `4`（GA4分析）+ `5`（Slack週報）+ `7`（学習エンジン） |
